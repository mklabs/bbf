Demo

Super simplified version of a Form Generator from Backbone.Models with the bare minimum functionnality I need. Actually, this is just a simple form html generation from given model attributes. The resulting html is designed to fit well with bootstrap's form layout.

This is still a Backbone.View, so it's expected that it should work in very much the same way.

I've made this to suit very specific needs. If you're looking into a robust and mature solution to generates Forms from Backbone.Models, you should definitely take a look at the really neat backbone-form. It'll go far beyond what this bbf thing is able to do, and it'll do in a much better way.

That beeing said, if you'd like to check out some more examples, you may want to take a look at this example page, see the annotated source code, or run the test suite.

Forms

Default functionnality

Forms views are given a model to work with on which the View will operate and generate the form's html. It'll use a template function that can be overriden by passing in a precompiled template function property.

Forms are able to generate text inputs, select lists, textareas and checkboxes.

View this example source »

Nested objects

When one of the model attributes is a nested object, the html is recursively generated. Each id / name attributes are created based on the current key and optional nested level key, taking the form of input-viewid-nested-key where viewid refers to the Backbone View's cid property.

View this example source »

Extending Backbone.Form

A Backbone.Form can be created and used right away, but they're still Backbone.View. So, it's expected that they should behave in very much the same way

That being said, we could create a custom Backbone.View extending from Backbone.Form and still be able to use properties like events, tagName or className. In addition to that, the custom template property could be used to pass in a precompiled template function to utterly customize the form output.

What about form validation?

I'd recommend putting the validation logic into your models, not inside the views, even if it is a form view. the serialize method is meant to return an object that is suitable to use with models' set method, regardless the nested object structure and "levels".

Optionnaly, the forms may (or may not) listen for error events on the provided model and provides some defaults error handling (such as displaying the error message in an .error box)

What about data-binding?

Not yet implemented, but hopefully, we'll get there. The basic idea is that whenever one of the fields' value changes, the according model attributes is udpated (and vice-versa). The change will probably happen with { silent: true } by default, but that'll certainly need to be configurable through options.
